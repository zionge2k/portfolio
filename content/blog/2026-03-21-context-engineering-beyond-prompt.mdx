## 문제

"프롬프트만 잘 쓰면 AI가 알아서 한다."

이 말은 반만 맞다. 여행 예약 에이전트에게 "다음 달 DevOps 컨퍼런스를 위해 파리에 호텔 예약해줘"라고 요청하면, 에이전트는 미국 켄터키주 파리의 Best Western을 예약한다. 위치를 "파리, 프랑스"로 고쳐 말하면 이번에는 1박 €900의 리츠 호텔을 예약한다.

첫 번째 실패는 프롬프트의 문제다 — 위치를 명시하지 않았다. 하지만 두 번째 실패는 프롬프트가 아니다. **회사 출장 정책이라는 컨텍스트가 에이전트에게 제공되지 않은 것**이 원인이다.

프롬프트를 아무리 정교하게 다듬어도, 에이전트가 필요한 정보를 갖고 있지 않으면 올바른 결과를 낼 수 없다.

## 해결책: 컨텍스트 엔지니어링

프롬프트 엔지니어링(Prompt Engineering)은 LLM에 전달하는 **지시문 자체**를 잘 작성하는 기술이다. 컨텍스트 엔지니어링(Context Engineering)은 추론 시점에 LLM이 보게 되는 **모든 것**을 설계하는 시스템 수준의 분야다.

| | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|---|---|---|
| **범위** | 지시문 하나 | 프롬프트 + 메모리 + RAG + 도구 + 상태 |
| **시점** | 작성 시(static) | 런타임(dynamic) |
| **결과** | 더 나은 질문 | 더 나은 시스템 |
| **비유** | 질문을 잘 하는 기술 | 질문에 필요한 자료를 준비하는 기술 |

프롬프트 엔지니어링은 컨텍스트 엔지니어링의 **부분집합**이다. 최종 프롬프트의 80%는 동적으로 조립된 콘텐츠(메모리, RAG 결과, 상태)이고, 20%만 정적 지시문이다.

## 구성 요소

컨텍스트 엔지니어링은 5가지 구성 요소를 오케스트레이션한다.

### 1. 메모리 (Memory)

| 유형 | 역할 | 예시 |
|------|------|------|
| **단기 메모리** | 컨텍스트 윈도우 내 대화 유지 | 긴 대화 요약, 이전 턴 참조 |
| **장기 메모리** | 벡터 DB로 과거 정보 검색 | 사용자 선호도, 과거 여행 기록 |

### 2. 상태 관리 (State Management)

다단계 작업에서 "지금 어디까지 했는지"를 추적한다. 항공편 예약 → 호텔 예약 → 공항 이동 수배까지 일련의 작업에서, 항공편 도착 시간이 호텔 체크인 시간에 영향을 주고, 이것이 다시 공항 이동 스케줄에 영향을 준다. **상태가 없으면 에이전트는 각 단계를 독립적으로 처리하고, 결과가 서로 맞지 않게 된다.**

### 3. RAG (Retrieval Augmented Generation)

에이전트를 동적 지식 소스에 연결한다. 핵심은 **전체 문서를 반환하지 않는다**는 점이다. 회사 출장 정책을 검색할 때 100페이지짜리 문서 전체가 아니라, "파리 지역 호텔 최대 허용 요금" 섹션만 선별해서 에이전트에게 전달한다.

시맨틱 매칭(의미 기반)과 키워드 매칭(텍스트 기반)을 결합한 하이브리드 검색이 표준이다.

### 4. 도구 (Tools)

LLM 자체로는 데이터베이스를 조회하거나 API를 호출할 수 없다. 도구가 그 격차를 해소한다.

```
도구 설명(Tool Description) 예시:

이름: check_travel_policy
설명: 회사 출장 정책에서 특정 도시의 호텔 최대 허용 요금을 조회한다.
사용 시점: 호텔 예약 전에 항상 호출해야 한다.
제약: 도시명은 영문으로 전달해야 한다.
```

**도구 설명의 품질이 에이전트의 도구 사용 능력을 결정한다.** "무엇을 하는지", "언제 사용하는지", "어떤 제약이 있는지"를 명확히 기술해야 한다.

### 5. 프롬프트 (Prompt)

프롬프트 엔지니어링의 4가지 핵심 기법은 여전히 유효하다:

- **역할 지정(Role Assignment)**: "보안 취약점을 검토하는 시니어 개발자"
- **Few-shot 예시**: 원하는 출력 형식을 2-3개 예시로 보여준다
- **Chain of Thought**: "단계별로 생각해 봅시다" — 성급한 결론 방지
- **제약 조건 설정**: "100단어 이내", "제공된 정보만 사용"

단, 이 프롬프트는 런타임에 동적 컨텍스트(메모리, RAG 결과, 상태)로 주입되어 최종 형태가 만들어진다.

## 실전: 컨텍스트 엔지니어링은 데이터 엔지니어링이다

한 개발자가 분석 에이전트의 신뢰성을 45%에서 86%로 개선한 사례가 있다. 40개 단위 테스트로 실패를 분류한 결과:

| 실패 원인 | 건수 | 비중 |
|----------|------|------|
| 데이터 모델 오류 (불명확, deprecated, 불완전) | 18 | 64% |
| 날짜 선택 모호성 | 6 | 21% |
| 테스트 자체 문제 | 4 | 14% |

**실패의 64%가 "에이전트에게 제공된 데이터의 품질" 문제였다.** 프롬프트가 아니라 컨텍스트가 문제였다. 데이터 모델의 모호성을 제거하고, 필드 설명을 추가하고, deprecated 항목을 정리하는 것만으로 21%포인트가 올랐다.

> 컨텍스트 엔지니어링은 결국 데이터 엔지니어링이다. 복잡한 시맨틱 레이어 없이도, 간단한 메타데이터와 규칙 파일만으로 높은 신뢰성을 달성할 수 있다.

## 핵심 정리

- **프롬프트 엔지니어링은 출발점이지 종착점이 아니다.** 프로덕션 AI 시스템에서 프롬프트 외의 컨텍스트가 결과 품질을 결정한다
- **컨텍스트 엔지니어링 = 메모리 + 상태 + RAG + 도구 + 프롬프트.** 5가지를 런타임에 동적으로 조립하는 것이 핵심이다
- **최종 프롬프트의 80%는 동적 콘텐츠다.** 정적 템플릿에 의존하면 한계가 있다
- **도구 설명의 품질 = 에이전트의 능력.** MCP의 tool description 설계가 여기에 해당한다
- **컨텍스트 엔지니어링은 데이터 엔지니어링이다.** 에이전트 실패의 대부분은 프롬프트가 아니라 데이터 품질 문제다
